\section{Dogfood Strategy}

Kungfu dogfood is not a demo. It is the path by which Kungfu learns what real
agent work management requires before public launch.

The initial dogfood workload is a private real-work system with long-running
tasks, local and remote agents, repository coordination, task ledgers, review
records, friction queues, trace evidence, and multi-machine timelines. Many of
these mechanisms currently exist as hand-built workflows around Git, files, and
logs. They work, but they also expose the cost of managing runtime facts
without a product designed for that purpose.

Kungfu should absorb the lessons, not import private terminology. The sequence
is:

\begin{enumerate}[leftmargin=*]
  \item import private real-work facts into Kungfu so the GUI can display them;
  \item let agents write runtime facts, traces, friction, and task state into
  Kungfu while the existing private workflow remains a backup source during the
  transition;
  \item identify which semantics are personal workflow choices and which belong
  in the generic Kungfu core;
  \item rebuild future private workflow mechanisms around the generic core
  where that reduces drift and token-heavy coordination;
  \item keep dogfood pressure high enough that Kungfu must become useful in
  daily work before public launch.
\end{enumerate}

The dogfood standard is load-bearing: if Kungfu cannot manage the work that is
building Kungfu, the product thesis has not been proven.
