\section{Architecture Direction}

Kungfu's architecture follows from the product thesis.

\subsection{Core Runtime Fact Layer}

At the center is a local runtime fact layer. It records durable facts about
work, agents, execution records, tasks, artifacts, cost, state transitions, and
observer timelines. The storage layer is not disposable cache. If Kungfu becomes a
source of truth for long-running work, storage needs the operational qualities
users expect from serious local infrastructure: verification, backup,
recovery, migration, and clear failure modes.

\subsection{Control Plane Interfaces}

Kungfu GUI and TUI are control planes for humans. They should make long-running
agent work visible without forcing the user to read logs or reconstruct
context manually. CLI, Python, and Node APIs are control planes for agents and
automation. They should expose the same facts in structured form.

The control plane is therefore not only a supervisor's panel over agent
activity. It is a shared work environment where humans and agents can discover
capabilities, inspect value evidence, understand constraints, and leave
reviewable records.

\subsection{Polyglot Membrane}

The product needs a polyglot membrane because the best ecosystem for each layer
is different:

\begin{itemize}[leftmargin=*]
  \item frontend and desktop surfaces benefit from Node, Electron, React, and
  the web ecosystem;
  \item agent integration benefits from Python and Node APIs;
  \item durable high-performance core surfaces benefit from C++ where needed.
\end{itemize}

The membrane is not accidental complexity. It lets Kungfu compress complexity
inside the product while exposing low-friction interfaces to humans and agents.

\subsection{Agent Supply Chain}

The public architecture separates five responsibilities that products often
rebuild inside one private control plane:

\begin{enumerate}[leftmargin=*]
  \item KFD-3 exposes value, constraints, choices, commands, Exit, and records
  so products can discover how to cooperate.
  \item Buildchain binds product-owned declarations to an exact source cut,
  build, artifacts, checks, and promotion evidence.
  \item KFD-2 lets a receiver assess those claims for a declared purpose while
  retaining residual risk and decision ownership.
  \item libkungfu and \texttt{.kungfu} preserve admitted work facts, Episodes,
  roots, exports, and recovery evidence; applications still own domain facts.
  \item the KFD Agent Hub profile carries bounded responsibility objects across
  independently owned products, with receiver-owned admission.
\end{enumerate}

The layers remain separable authorities. Buildchain does not invent product
facts or make the trust decision. KFD does not become a central Hub, identity
provider, database, or transport. Kungfu's JSON surfaces are edge projections,
not a second authoritative data plane.

\subsection{Extension Model}

Kungfu extensions should let other users define their own work semantics
without forking the product. A research lab, operations team, developer,
writer, or personal user may need a different semantic model. The core should
support these variations through explicit schemas, extension packages, and
capability declarations.
