\section{Roadmap}

The roadmap should be judged by whether Kungfu becomes indispensable for real
daily work, not by whether it checks a list of fashionable AI features.

\subsection{MVP Gate}

The MVP should not be only an LLM tracer. A tracer can be useful, but it is a
secondary function under real-world agent work management. The MVP should
prove that a normal agent user can install Kungfu and get a local control plane
for persistent work.

The practical MVP gate is:

\begin{itemize}[leftmargin=*]
  \item Kungfu can manage real work units through GUI/TUI plus CLI/API;
  \item agents can bootstrap themselves into the product without reading a long
  manual;
  \item local task/fact state is durable enough to trust for private dogfood;
  \item tracing and cost management are integrated as natural subfunctions;
  \item Buildchain/KFD release evidence proves the public surfaces are not
  drifting;
  \item the product is useful enough to become a daily resident application for
  an initial real-work user.
\end{itemize}

\subsection{Dogfood Stage}

The first serious dogfood profile is deliberately personal and opinionated,
which makes it valuable as a stress test. If Kungfu can absorb long-running
work units, durable task ledgers, review records, friction queues, traces,
agent bootstrap, and multi-machine timelines from real daily use, then the core
is likely strong enough to support more general profiles.

\subsection{Public Release Stage}

Public release requires a stable installation path, a low-friction first-run
experience, agent-readable onboarding, cross-platform behavior, migration and
backup boundaries, and enough documentation for people and agents to understand
what the product does.

\subsection{Ecosystem Stage}

After the core is proven, Kungfu should become a platform for extensions. Users
and organizations should be able to build their own work models, dashboards,
agent integrations, and domain-specific capabilities on top of the core.
